# AI와 페어 디버깅하기 — Claude Code로 JPA 매핑 버그를 잡은 이야기

## 왜 AI와 페어 디버깅인가

버그를 만나면 보통 이런 순서를 밟는다:

1. 에러 로그를 읽는다
2. 구글링한다
3. Stack Overflow를 뒤진다
4. 관련 코드를 뒤져본다
5. 가설을 세우고 검증한다

혼자 하면 3번과 4번에서 시간이 많이 든다. 특히 에러의 원인이 "내 코드의 구조적 불일치"일 때는 — 검색으로는 잘 안 나온다. 내 프로젝트의 맥락을 알아야 하기 때문이다.

AI 코딩 에이전트(여기서는 Claude Code)와 페어 디버깅하면 이 과정이 달라진다. AI에게 에러 상황을 설명하면, **내 코드베이스를 직접 탐색하면서** 가설을 세우고 검증해 준다. 검색 결과를 읽고 내 상황에 맞게 해석하는 과정을 건너뛸 수 있다.

사이드 프로젝트 [TriAgain(작심삼일 크루)](https://github.com/triagain)을 만들면서 겪은 실제 디버깅 사례로 이 과정을 정리한다.

## 사례: JPA @IdClass 매핑 버그

> 이 사례의 기술적 분석은 [JPA @IdClass 복합 PK의 함정](./blog-jpa-idclass-pitfall.md)에 자세히 정리했다.
> 여기서는 AI와 어떻게 협업했는지에 초점을 맞춘다.

### 상황

크루 생성 API를 호출하면 `crew_members` 테이블 INSERT에서 에러가 발생했다.

```
ERROR: null value in column "id" of relation "crew_members" violates not-null constraint
```

Hibernate가 생성한 SQL에서 `id` 컬럼이 빠져 있었다. 원인은 DB 스키마(서로게이트 PK)와 JPA 엔티티(`@IdClass` 복합 PK)의 불일치였다.

### 디버깅 흐름

#### 1단계: 에러 상황 + 컨텍스트 공유

처음에 한 일은 단순하다. 에러 로그와 Hibernate SQL을 그대로 붙여넣었다.

> **나**: 크루 생성 API 호출하면 INSERT에서 에러가 나. 에러 로그랑 Hibernate SQL 같이 볼래?

중요한 건 **에러 메시지만 던지지 않은 것**이다. Hibernate가 생성한 SQL도 함께 공유했다. 이 SQL이 없었으면 AI도 "INSERT문을 보여달라"고 한 번 더 물어봤을 것이다.

#### 2단계: AI의 가설 생성 + 코드 탐색

Claude는 INSERT SQL을 보자마자 핵심을 짚었다.

> **Claude**: INSERT SQL을 보면 `id` 컬럼이 아예 빠져 있습니다.
> `crew_members` 테이블의 DDL과 `CrewMemberJpaEntity`의 `@Id` 매핑을 비교해 볼까요?

여기서 AI의 첫 번째 강점이 나왔다. 사람이라면 여러 가능성을 떠올리며 하나씩 확인했을 텐데, AI는 에러 메시지("null value in column id")와 SQL(id 컬럼 부재)을 바로 연결했다.

schema.md를 공유하자 Claude는 즉시 불일치를 발견했다:

> **Claude**: DB 스키마에는 `id`가 서로게이트 PK인데,
> `CrewMemberJpaEntity`는 `@IdClass(CrewMemberId.class)`로 복합 PK를 선언하고 있습니다.
> **DB의 PK 전략과 JPA의 PK 전략이 불일치합니다.**

그리고 스스로 다음 행동을 제안했다:

> **Claude**: 프로젝트의 다른 엔티티들도 한번 확인해 볼게요.

#### 3단계: 전수 스캔 — 사람이 하기 번거로운 작업

이 부분이 AI 페어 디버깅의 하이라이트였다. Claude Code는 프로젝트 내 모든 JPA 엔티티 파일을 탐색해서 `@Id` 패턴을 비교했다.

결과:

| 엔티티 | PK 전략 |
|--------|---------|
| `CrewJpaEntity` | 서로게이트 `@Id` |
| `UserJpaEntity` | 서로게이트 `@Id` |
| `ChallengeJpaEntity` | 서로게이트 `@Id` |
| `VerificationJpaEntity` | 서로게이트 `@Id` |
| `UploadSessionJpaEntity` | 서로게이트 `@Id` |
| **`CrewMemberJpaEntity`** | **`@IdClass` 복합 PK** |

**하나만 다른 패턴을 쓰고 있었다.** 사람이 수동으로 6개 파일을 열어서 `@Id` 어노테이션을 비교하는 건 가능하지만 번거롭다. AI는 몇 초 만에 끝냈다.

#### 4단계: 수정안 제시 → 리뷰 후 적용

Claude는 수정이 필요한 파일 4개를 정리해서 제시했다:

1. `CrewMember` 도메인 모델 — `id` 필드 + `IdGenerator.generate("CRMB")` 추가
2. `CrewMemberJpaEntity` — `@IdClass` 제거, 서로게이트 PK로 변경
3. `CrewMemberJpaRepository` — 타입 파라미터 변경
4. `CrewMemberId` 클래스 — 삭제

수정안이 프로젝트의 기존 패턴과 정확히 일치했기 때문에, 리뷰 후 바로 적용했다.

## AI 페어 디버깅 워크플로우

위 사례를 일반화하면 이런 워크플로우가 된다.

### Step 1. 에러 상황 + 컨텍스트를 함께 공유한다

에러 메시지만 던지지 말고, 관련 정보를 함께 준다.

| 공유하면 좋은 것 | 예시 |
|-----------------|------|
| 에러 로그 전문 | 스택 트레이스, SQL 로그 |
| 재현 조건 | "크루 생성 API 호출 시" |
| 관련 코드/설정 | 엔티티 클래스, DDL, 설정 파일 |
| 이미 시도한 것 | "구글링해서 X를 해봤는데 안 됐어" |

**팁**: 프로젝트 내 문서(schema.md, architecture.md 등)가 있으면 컨텍스트 공유가 훨씬 수월하다. 이번 사례에서도 schema.md가 있었기에 DDL을 일일이 찾을 필요 없이 문서를 공유하는 것만으로 충분했다.

### Step 2. AI가 가설을 세우고 코드를 탐색한다

AI에게 "왜 이런 건지 찾아봐"라고 하면, 코드베이스를 탐색하면서 가설을 세운다. 이 과정에서 AI가 잘하는 것들:

- **패턴 매칭**: 에러 메시지의 키워드에서 관련 코드를 찾아간다
- **비교 분석**: 동일한 패턴을 쓰는 다른 코드와 비교해서 차이점을 찾는다
- **전수 스캔**: 프로젝트 전체에서 특정 어노테이션이나 패턴을 검색한다

### Step 3. 사람이 판단하고, AI가 수정안을 제시한다

AI가 원인을 찾으면 수정 방향을 제안한다. 이때 사람이 해야 할 것:

- 수정안이 프로젝트 컨벤션에 맞는지 확인
- 사이드 이펙트가 없는지 판단
- 여러 선택지가 있을 때 어떤 방향으로 갈지 결정

### Step 4. 리뷰 후 적용

AI가 작성한 코드를 리뷰하고 적용한다. 코드 리뷰는 반드시 사람이 해야 하는 과정이다.

## 효과적인 AI 디버깅 프롬프트 팁

### 좋은 프롬프트

```
크루 생성 API 호출 시 crew_members INSERT에서 에러가 나.
에러 로그:
[에러 로그 붙여넣기]

Hibernate SQL:
[SQL 붙여넣기]

schema.md에 정의된 테이블 구조:
[DDL 붙여넣기]
```

- 에러 상황, 로그, 관련 코드를 한 번에 제공
- AI가 추가 질문 없이 바로 분석에 들어갈 수 있다

### 아쉬운 프롬프트

```
INSERT 에러 나는데 왜 그래?
```

- 어떤 테이블인지, 어떤 에러인지 알 수 없다
- AI가 "에러 로그를 보여주세요", "어떤 엔티티인가요?" 등 추가 질문을 해야 한다
- 왕복이 늘어나고 디버깅 속도가 느려진다

### 팁 요약

| 원칙 | 설명 |
|------|------|
| **에러 로그 전문을 붙여라** | 부분 발췌보다 전문이 낫다. AI가 놓치는 단서가 줄어든다 |
| **관련 코드를 함께 줘라** | 에러 로그만으로는 원인을 좁히기 어렵다 |
| **이미 시도한 걸 말해라** | 같은 제안을 반복하는 걸 막는다 |
| **"왜"보다 "뭐가 달라?"를 물어라** | "왜 안 되지?"보다 "다른 엔티티는 되는데 이것만 안 돼. 뭐가 다를까?"가 더 좋은 결과를 낸다 |

## AI 페어 디버깅의 한계와 주의점

### AI가 잘하는 것

- **에러 메시지 해석**: 에러 로그에서 핵심 키워드를 빠르게 추출한다
- **패턴 비교**: 프로젝트 내 유사한 코드와 비교해서 불일치를 찾는다
- **전수 스캔**: 수십 개 파일에서 특정 패턴을 검색하는 건 AI가 압도적으로 빠르다
- **수정안 작성**: 기존 패턴에 맞춰 수정 코드를 생성한다

### AI가 못하는 것 (사람이 해야 하는 것)

- **비즈니스 컨텍스트 판단**: "이 테이블에 복합 PK가 정말 필요한가?"는 도메인 지식이 필요하다
- **아키텍처 결정**: 여러 수정 방향이 있을 때 어떤 것을 선택할지는 사람의 판단이다
- **사이드 이펙트 예측**: 수정이 다른 기능에 미치는 영향은 사람이 전체 시스템을 고려해서 판단해야 한다
- **최종 코드 리뷰**: AI가 생성한 코드는 반드시 사람이 리뷰해야 한다

### 주의점

- **AI의 답을 무조건 신뢰하지 마라** — AI도 틀릴 수 있다. 특히 프로젝트 고유의 컨벤션이나 히스토리를 모를 수 있다
- **컨텍스트가 부족하면 AI도 헤맨다** — 정보를 충분히 주지 않으면 엉뚱한 방향으로 갈 수 있다
- **디버깅을 통째로 맡기지 마라** — "이거 고쳐줘"보다 "이런 에러가 나는데 원인을 같이 찾아보자"가 낫다. 과정을 함께 따라가야 배움이 있다

## 마무리

AI와 페어 디버깅은 "혼자 삽질하는 시간"을 크게 줄여준다. 특히 **내 코드베이스의 구조적 문제**를 찾을 때 강점이 두드러진다. 구글링으로는 "내 프로젝트에서 CrewMember만 @IdClass를 쓰고 있다"는 걸 알 수 없지만, AI는 코드를 직접 탐색해서 찾아낸다.

다만 AI는 도구이지 판단의 주체가 아니다. 에러 상황을 제공하고, AI의 분석을 검증하고, 최종 결정을 내리는 건 개발자의 몫이다.

**사람이 컨텍스트를 주고, AI가 탐색하고, 사람이 판단한다** — 이 역할 분담이 AI 페어 디버깅의 핵심이다.
